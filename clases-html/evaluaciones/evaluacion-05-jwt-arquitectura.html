Requisitos Previos

REQUIERE E4 (SQLAlchemy ORM) completada. Se agregará lógica de negocio, autenticación JWT y arquitectura en capas sobre los modelos ya creados.

Clases Relacionadas

Estas clases preparan los conocimientos necesarios para esta evaluación:

  • Clase 04: CRUD SQLAlchemy Session, add, commit, query, delete
  • Clase 05.1: Patrón Repository Repositorios genéricos e inyección de dependencias
  • Clase 05.2: Clean Architecture + JWT Separación de capas, autenticación JWT y protección de rutas

Objetivo de Aprendizaje

Construir la API funcional de TaskFlow con operaciones CRUD completas, autenticación JWT y arquitectura en capas. Todo debe estar desacoplado: routers llaman a servicios, servicios usan repositorios y los modelos SQLAlchemy nunca exponen directamente la base de datos a los endpoints.

Modalidad de Trabajo

Individual o en parejas (máximo 2 personas). Debe ser el mismo equipo de E1-E4.

Formato de entrega: Repositorio GitHub con la API ejecutable. README.md debe incluir instrucciones para levantar el servidor, aplicar migraciones y probar los endpoints (puede incluir ejemplos con curl).

Requisitos Técnicos Detallados

1. CRUD SQLAlchemy (25%)

Descripción: Endpoints completos para crear, leer, actualizar y eliminar las entidades principales.

Entidades obligatorias con CRUD:
  • Usuario: registro, login, obtener perfil, listar usuarios
  • Proyecto: crear, listar por usuario, editar, eliminar
  • Tarea: crear, listar por proyecto, editar estado, eliminar
Requisitos técnicos:
  • Uso correcto de Session: add, commit, rollback en errores, refresh
  • Consultas con filter, .first(), .all(), .count()
  • No exponer contraseñas en las respuestas JSON
Ejemplo mínimo de endpoint: 
@router.post("/proyectos", response_model=ProyectoOut)
def crear_proyecto(
    data: ProyectoCreate,
    db: Session = Depends(get_db),
    usuario: Usuario = Depends(get_current_user)
):
    proyecto = Proyecto(nombre=data.nombre, usuario_id=usuario.id)
    db.add(proyecto)
    db.commit()
    db.refresh(proyecto)
    return proyecto
2. JWT Authentication (25%)

Descripción: Implementar registro, login y protección de rutas sensibles usando tokens JWT.

Requisitos obligatorios:
  • Endpoint POST /auth/register: crea usuario con contraseña hasheada (passlib / bcrypt)
  • Endpoint POST /auth/login: valida credenciales y retorna access_token (python-jose)
  • Dependencia get_current_user que decodifique el JWT y obtenga el usuario de la base de datos
  • Proteger rutas de creación/edicion/eliminación con Depends(get_current_user)
  • Usar OAuth2PasswordBearer para documentación automática en Swagger UI
Verificación rápida: 
# Obtener token
curl -X POST http://localhost:8000/auth/login \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "username=ana&password=secret123"

# Usar token
curl http://localhost:8000/proyectos \
  -H "Authorization: Bearer <token>"
3. Arquitectura en Capas (20%)

Descripción: Separar responsabilidades en carpetas claras y usar DTOs con Pydantic v2.

Capas obligatorias:
  • routers/: define endpoints y dependencias de FastAPI
  • services/: lógica de negocio (validaciones, reglas, orquestación)
  • repositories/: acceso a datos con SQLAlchemy
  • models/: entidades SQLAlchemy
  • schemas/: DTOs Pydantic v2 para entrada/salida de datos
Requisitos de schemas:
  • Al menos un schema Create, uno Out y uno Update por entidad principal
  • Usar ConfigDict(from_attributes=True) para mapeo con ORM
  • Los routers deben devolver schemas, no modelos SQLAlchemy directamente
4. Manejo de Errores (15%)

Descripción: Responder con códigos HTTP correctos y mensajes consistentes.

  • 404 Not Found cuando no existe un recurso
  • 401 Unauthorized cuando falta o es inválido el token JWT
  • 400 Bad Request para validaciones de negocio (ej: email duplicado)
  • No devolver 500 por errores de validación predecibles
5. Dependency Injection (10%)

Descripción: Inyectar sesiones de base de datos y repositorios usando Depends de FastAPI.

  • get_db() como generador de Session
  • Repositorios inyectados en los servicios o routers (ej: repo: ProyectoRepo = Depends(get_proyecto_repo))
  • Cerrar sesiones correctamente con finally: db.close()
6. POO Avanzada (5%)

Descripción: Aplicar patrones de diseño y principios POO.

  • Repository pattern: interfaz genérica Repositorio[T] con ABC
  • Herencia de schemas base si aplica (ej: BaseSchema con model_config)
  • Encapsulamiento: atributos privados o uso de properties cuando sea necesario

Penalizaciones
  • -20% si no hay autenticación JWT funcional
  • -20% si no hay separación de capas (todo en main.py)
  • -15% si no hay schemas Pydantic v2
  • -10% por cada día de retraso
  • Descalificación si se entrega sin repositorio GitHub o la API no compila
